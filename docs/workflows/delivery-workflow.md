# 納品ワークフロー - 対話形式での課題解決

## 概要

`/qa full` で生成された納品チェックリストを基に、対話形式で一つずつ課題を解決していくワークフローです。

**推奨**: 新規チャットで実行（コンテキストオーバーフロー対策）

---

## ワークフロー全体図

```
┌─────────────────────────────────────────────────────────────────┐
│ Phase 1: QA実行（メインチャット）                               │
└─────────────────────────────────────────────────────────────────┘
                           ↓
                    /qa full 実行
                           ↓
          reports/delivery-checklist-YYYYMMDD.md 生成
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 2: 課題修正（新規チャット）                               │
└─────────────────────────────────────────────────────────────────┘
                           ↓
                  新規チャット起動
                           ↓
              /delivery fix 実行（繰り返し）
                           ↓
        ┌──────────────┬──────────────┬──────────────┐
        │              │              │              │
   Critical修正    High修正    Medium修正    Low修正（任意）
        │              │              │              │
        └──────────────┴──────────────┴──────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Phase 3: 最終確認（メインチャットor新規）                       │
└─────────────────────────────────────────────────────────────────┘
                           ↓
                   /qa verify 実行
                           ↓
               /delivery report 実行
                           ↓
              クライアント提出・納品
```

---

## Phase 1: QA実行（メインチャット）

### 1-1. 完全QA実行

```bash
/qa full
```

**実行内容:**
- 機械的チェック（Lint/Build/Links/Images等）
- 人間的レビュー（PHP/SCSS/JS）
- 検出された課題の修正
- 納品チェックリスト生成

**生成物:**
- `reports/delivery-checklist-YYYYMMDD.md` - 社内用詳細チェックリスト
- `reports/delivery report-client-YYYYMMDD.md` - クライアント提出用レポート
- `.claude/reviews/*.md` - 詳細レビュー結果

### 1-2. チェックリストの確認

生成されたチェックリストで以下を確認:
- **Critical課題**: 即座に対応必須（赤🔴）
- **High課題**: 納品前に対応推奨（黄⚠️）
- **Medium/Low課題**: 納品後の改善として対応可（青💡）

---

## Phase 2: 課題修正（新規チャット推奨）

### 2-1. 新規チャット起動

**理由:**
- Phase 1のコンテキストが大きい（レビュー結果等）
- Phase 2で複数の修正を行うとさらにコンテキストが膨れる
- 新規チャットで進めることでクリーンな環境を維持

**チャット起動時の準備:**
1. 新規チャット起動
2. プロジェクトフォルダを開いた状態で開始
3. 最初のメッセージ: `/delivery fix`

### 2-2. 対話形式での修正

```bash

/delivery fix
```

**実行例:**

### 2-3. 特定の課題を指定

```bash

```

## Phase 3: 最終確認

### 3-1. 最終検証

```bash
/qa verify
```

**実行内容:**
- 修正後の全体チェック
- 問題が0件であることを確認
- ビルド成功確認

### 3-2. 納品レポート更新

```bash
/delivery report
```

**実行内容:**
- チェックリストの進捗を反映
- クライアント提出用レポートを更新
- 品質スコアを更新
- 納品日を更新

**生成物:**
- `reports/delivery report-client-YYYYMMDD.md` - 更新版

### 3-3. 手動確認項目の実施

チェックリストの手動確認項目を担当者が実施:
- [ ] PC表示確認（Chrome/Safari/Firefox/Edge）
- [ ] SP表示確認（実機テスト）
- [ ] タブレット表示確認
- [ ] 誤字脱字チェック
- [ ] リンク先ページの内容確認
- [ ] Google Analytics/GTM動作確認
- [ ] クライアント最終確認

### 3-4. クライアント提出

`reports/delivery report-client-YYYYMMDD.md` をクライアントに提出。

## エージェント自動起動ルール

`/delivery fix` は課題の種類に応じて自動的に適切なエージェントを起動します:

| 課題カテゴリ | 起動エージェント | 対応内容 |
|-------------|-----------------|---------|
| **SCSS (Safe)** | `code-fixer` | カラーコード変数化、空ブロック削除、プロパティ順序修正 |
| **SCSS (Risky)** | `wordpress-professional-engineer` | HTML構造を伴うSCSS変更（container/BEMネスト等） |
| **PHP/WordPress** | `code-fixer` | エスケープ処理、ACFチェック、セキュリティ対策 |
| **SEO/OGP** | `code-fixer` | OGPタグ実装、メタタグ追加 |
| **JavaScript** | `code-fixer` | console.log削除、未使用コード削除、メモリリーク対策 |
| **HTML構造** | `wordpress-professional-engineer` | セマンティックHTML化、article要素化 |

## チェックリストの更新方法

修正完了後、チェックリストは自動的に更新されます:

### 修正前
```markdown
- [ ] **container-001** (Critical): コンテナ規約違反
  - ファイル: `src/scss/object/projects/company/_p-global-expansion.scss:28`
  - 推定時間: 30分
```

### 修正後
```markdown
- [x] **container-001** (Critical): コンテナ規約違反 ✅
  - ファイル: `src/scss/object/projects/company/_p-global-expansion.scss:28`
  - 修正日時: 2025-12-15 16:30
  - 担当: code-fixer
```

## ベストプラクティス

### ✅ DO（推奨）

1. **新規チャットで実行**
   - Phase 1 (QA) と Phase 2 (修正) は別チャットで
   - コンテキスト管理が容易

2. **優先度順に対応**
   - Critical → High → Medium → Low の順
   - `/delivery fix` は自動的に優先度順に提示

3. **こまめにコミット**
   - 各課題修正後に `git commit`
   - 問題があればロールバック可能

4. **検証を怠らない**
   - 修正後は必ず `npm run qa:check`
   - 新たな問題が発生していないか確認

### ❌ DON'T（非推奨）

1. **並列修正**
   - 複数の課題を同時に修正しない
   - コンフリクトのリスクあり

2. **チェックリストの手動編集**
   - `/delivery fix` が自動更新するので手動編集不要
   - 手動編集すると進捗管理が狂う

3. **検証スキップ**
   - 「小さな修正だから」と検証をスキップしない
   - 予期せぬ副作用がある可能性

## トラブルシューティング

### Q: `/delivery fix` が「チェックリストが見つかりません」と言う

**A:** チェックリストのパスを明示的に指定:
```bash
/delivery fix --checklist reports/delivery-checklist-20251215.md
```

### Q: 修正後に新たな問題が発生した

**A:** 該当のコミットをロールバック:
```bash
git reset --hard HEAD~1
```

問題を再度分析してから修正。

### Q: エージェントが起動しない

**A:** 手動で専門エージェントを起動:
```bash
# 例: SCSS問題の場合
Task tool: subagent_type=code-fixer
prompt: "reports/delivery-checklist-20251215.md の container-001 を修正してください"
```

### Q: コンテキストが溢れてきた

**A:** 新しいチャットを起動して再開:
1. 現在の進捗を確認: `/delivery fix --status`
2. 新規チャット起動
3. `/delivery fix` で続きから再開（チェックリストは永続化されている）

## まとめ

このワークフローにより:
- ✅ 課題を見落とさない（チェックリスト管理）
- ✅ 優先度順に効率的に対応（自動ソート）
- ✅ 専門エージェントが自動起動（手間なし）
- ✅ 進捗が可視化される（モチベーション維持）
- ✅ コンテキストオーバーフローを回避（新規チャット）

**推奨フロー:**
1. メインチャット: `/qa full` でチェックリスト生成
2. 新規チャット: `/delivery fix` で課題を一つずつ解決
3. 完了後: `/qa verify` + `/delivery report` で最終確認
4. クライアント提出・納品
