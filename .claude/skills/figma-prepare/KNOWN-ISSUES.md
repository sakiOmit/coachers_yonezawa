## OPEN Issues

### Issue 21: Phase 2（リネーム）と Phase 3（グルーピング）の実行順序を入れ替える

- **Phase**: 2, 3
- **優先度**: 高
- **概要**: 現在の順序（リネーム → グルーピング）は非効率。グルーピングを先に行い、確定した構造に対してリネームすべき。
- **課題**:
  - リネーム後にグルーピングすると、新しく作成されたグループフレームの命名が漏れる
  - グルーピングで構造が変わるため、リネーム結果が「中間状態」になる
  - 実質、グルーピング後にもう一度リネームが必要になってしまう
- **正しい順序**:
  - Phase 2: グルーピング + セクショニング（構造確定）
  - Phase 3: セマンティックリネーム（確定構造に対して一括命名）
- **根拠**:
  - `detect-grouping-candidates.sh` はノード名に依存しない（type, 座標, 子構造のみ使用）→ リネーム前でも動作する
  - リネームロジックはグループ化後のコンテキスト（子構造）を利用できるため、精度が向上する
  - 新グループフレーム含め全ノードを一括でリネームできる
- **影響範囲**:
  - SKILL.md: フロー図、Phase 2/3 の説明全体を入れ替え
  - テスト: Phase 番号参照の更新
  - `--phase 2` / `--phase 3` の意味が変わる（破壊的変更）
  - キャリブレーションデータ・品質基準の Phase 番号参照
- **移行案**:
  - Phase 番号を振り直す（Phase 2 = グルーピング、Phase 3 = リネーム）
  - または内部実行順序のみ変更し、Phase 番号は維持（混乱のリスクあり）

### Issue 22: Phase 3 グルーピング精度が実期待に達していない

- **Phase**: 3
- **優先度**: 高
- **概要**: 募集一覧ページ (1:4) のフラットな8子要素が、関連要素ごとに適切にグルーピングされるべきだが、現状の Stage A ヒューリスティックでは不十分。

  ```
  募集一覧
  ├── Group（タブ + カード一覧）
  │   ├── 2. 募集区分タブ
  │   └── 3. 募集カード一覧（12件）
  ├── 3. 募集カード一覧（0件）

  ├── Group（ヒーロー画像 + 見出し + パンくず）
  ```
